 코드 리뷰 — `/phase`(AI 국면 판정) 검증 루프

> 리뷰 대상: 최신 커밋 `b1cebe3` "feat: AI 국면 판정(/phase) 검증 루프 — KV 기록 + 익일 채점 + 성과 패널"
> 리뷰 방식: 10개 독립 탐지 앵글(정확성 5 + 정리 3 + 고도 1 + 컨벤션 1) + 갭 스윕 + 직접 소스 대조 검증
> 효력 수준: xhigh(recall 우선 — 놓친 실버그가 더 위험하므로 의심되면 surface)

## 총평

기능은 **기계적으로 안전하다.** 이미 운영 중인(그리고 최근 "실버그 11건"으로 한 번 단단해진)
`ai_click_eval.py` / aipred 기록 패턴을 충실히 복제했고, KV·평가·푸시의 모든 가드(손상파일 `.corrupt`
백업, 25일 만료, `close>0` 보장, 비원자성 갭 자가치유, git 락 직렬화)가 그대로 보존돼 있다.

**크래시·데이터 손실·빌드 깨짐 수준의 버그는 없다.** 발견된 항목은 두 갈래다.

1. **대시보드가 무엇을 보고하는가** — 이 기능의 존재 이유인 "신뢰도 높을수록 맞나" 보정(calibration)을
   왜곡·은폐하는 통계·UX 문제.
2. **유지보수성** — 이 3번째 클론이 더하는 중복.

CLAUDE.md 컨벤션 위반은 없다(통계 격리 `score_raw` 무관·자동튜닝 X, Python 표준라이브러리만, repository
SSOT 경유, install_cron 멱등, 한국 색 관례 준수). `DISCLAIMER` 리터럴은 시드 JSON과 바이트 단위로 일치.

## 발견 항목 (심각도 높은 순)

### 🟠 통계·UX 정확성

#### 1. 중립만 평가됐을 때 데이터가 통째로 숨겨짐
- **파일/라인:** `web/components/performance/PhasePerformancePanel.tsx:21`
- **요지:** 빈 상태 게이트가 `n === 0`(방향 표본=재매집+분산만)이라, 성숙한 표본이 전부 중립이면
  본문은 "데이터 수집 중"을 띄우면서 **항상 렌더되는** 헤더(line 18)는 "방향 표본 0건 (중립 N 별도)"라고
  말한다 — 서로 모순.
- **재현:** 초기 수집 단계에서 클릭된 몇 종목이 전부 중립 판정. phase_eval이 `total_n=5, n=0`을 기록.
  패널은 아무것도 채점 안 된 것처럼 "수집 중"을 보여주는데, 헤더는 중립 5건이 있다고 광고. 중립의
  rose_rate·avg_return과 최근 채점 목록(recent)이 재매집/분산 표본이 생길 때까지 안 보임.

#### 2. `confidence=50` 기본값이 보정 구간을 오염
- **파일/라인:** `scripts/phase_eval.py:225` (근원: `web/lib/stock/phase.ts:90`)
- **요지:** `buildPhaseAnalysis`는 LLM이 confidence를 누락/비유한 값으로 줄 때마다 **50**으로 기본값을
  채운다(`phase.ts:90`). 이 합성 50들이 `(0,60)` 구간에 섞여 들어가, 정작 이 패널이 만들려는
  "신뢰도 높을수록 맞나" 곡선을 편향시킨다.
- **재현:** Kimi가 쓸만한 confidence 없이 JSON을 반환(흔함) → recordPhase가 50을 적재 → 모든 해당
  표본이 `(0,60)`에 안착 → 그 구간 적중률은 실제 저신뢰 판정이 아니라 "모델 침묵" 케이스를 반영.
  경고 없이 헤드라인 보정 곡선이 오염됨.

#### 3. `평균 수익`(avg_return)이 `min_n` 게이트를 우회
- **파일/라인:** `web/components/performance/PhasePerformancePanel.tsx:60`
- **요지:** 국면별 행에서 `hit_rate`(line 57)와 `rose_rate`(line 59)는 둘 다 `b.valid`(n≥min_n)
  뒤에 숨는데, `평균 수익`만 게이트 없이 raw로 렌더된다.
- **재현:** 재매집 표본 3건일 때 — 방향 적중률은 "수집 중", 익일 상승률은 "—"(둘 다 게이트)인데
  평균 수익은 3거래 기준 "+4.2%"를 그대로 노출. min_n이 억제하려는 바로 그 소표본 단정이 한 컬럼으로 샌다.

#### 4. 보합일(flat day) 채점 비대칭
- **파일/라인:** `scripts/phase_eval.py:160`
- **요지:** `rose = nb["close"] > entry`가 strict 부등호라, 익일 종가가 신호일 종가와 정확히 같으면
  (rose=False) **분산→적중(True), 재매집→미적중(False)** 으로 채점된다.
- **재현:** 상한가/저유동 종목이 익일 신호일 종가에 정확히 마감(드물지 않음) → 분산은 "안 올랐다"로
  적중을 챙기고 재매집은 미적중. 표본 전체에서 분산 적중률을 부풀리고 재매집을 깎아 by_phase·
  confidence_bands를 왜곡. (정의상 선택이긴 하나 두 방향 phase 간 비대칭이 실재.)

#### 5. phase_eval.py = ai_click_eval.py의 3번째 사실상 복제본

  KV 키·채점·perf 형태만 빼면 문자 단위로 동일(~230/307 라인 중복).
- **비용:** 공용 루프에 대한 모든 수정(kv_get 토큰 우선순위, 25일 만료, `.corrupt` 백업, hgetall
  list-vs-dict 파싱, pull-rebase 재시도)을 이제 3개 파일에 넣어야 한다. track_eval은 이미 드리프트
  (kv_members vs kv_get)를 보인다. 하나라도 놓치면 한 eval이 조용히 갈라진다 — 리포가 방금 "실버그
  11건"으로 고친 바로 그 드리프트 부류. **파라미터화한 eval 루프(KV 네임스페이스 + score_fn +
  perf_builder)로 추출** 권고.

#### 6. recordPhase 트리플렛이 `/ai` 라우트와 중복
- **파일/라인:** `web/app/api/stock/[code]/phase/route.ts:42`

  네임스페이스 'phase'/'aipred'와 페이로드만 다르게 복붙됨.
- **비용:** `PRED_TTL_SEC`(90일)와 `{ns}:dates` 명명이 2개 라우트에 산재. TTL 변경이나 HSETNX→pipeline

#### 7. 패널·타입이 AiClickCalibrationPanel과 중복
- **파일/라인:** `web/components/performance/PhasePerformancePanel.tsx:1` (+ `web/types/performance.ts`)
- **요지:** 패널 122라인이 섹션 셸 + n===0 빈 상태 + valid/"수집 중" 게이트 + min_n 구간표 + recent
  목록 + disclaimer 푸터를 처음부터 재구현. types도 `*Band`/`*Recent`/`*Performance` 형태를 3중 반복.
- **비용:** 패널 122라인 중 ~70라인이 재사용 가능한 scaffolding인데 공용 셸/BandTable/EmptyState가

#### 8. cron이 야간 git 푸시 4회 → Vercel 재빌드 4회
- **파일/라인:** `scripts/install_cron.sh:10`

  ai_click(17:35)+phase(17:37)가 ~17분 안에 별도 커밋·푸시 4회, 각자 flock 경합 + 각자 Vercel 재빌드 유발.
- **비용:** 개념상 하나인 야간 eval 배치에 재빌드 4회 — CLAUDE.md가 명시적으로 지키는 무료 한도

### 🟢 효율·사소

#### 9. recordPhase가 응답 경로를 블로킹
- **파일/라인:** `web/app/api/stock/[code]/phase/route.ts:60`
- **요지:** recordPhase가 NextResponse.json 전에 inline await되며 Upstash 라운드트립 3회를 순차 발생.
  KV 기록은 부가 작업인데 사용자 응답을 불필요하게 막는다(`/ai`와 동일 — 기존 패턴).
- **재현:** CDN 미스마다 5~20초 Kimi 호출 + ~90~240ms 직렬 KV RTT(느린 KV 사고 시 최대 3×5초)를
  바이트 전에 대기. 분석은 이미 계산 완료 — `waitUntil`/`after()`로 지연하거나 `/pipeline` 1콜로
  묶으면 즉시 반환(에러는 이미 삼키므로 동작 불변).

#### 10. `valid` 필드는 파생 가능한 상태
- **파일/라인:** `scripts/phase_eval.py:215`
- **요지:** `_by_phase`의 `valid: len(grp) >= MIN_N`(그리고 동일한 `_conf_bands` line 230)은
  파생 상태 — `n`과 `min_n`이 이미 JSON에 실려 있고 패널도 이미 min_n을 소비한다.
- **비용:** 7개 셀(by_phase 3 + bands 4)에 저장돼 자기 `n`과 어긋날 수 있는 bool. 제거하고 단일
  소비처에서 `b.n >= min_n`로 계산하면(min_n 이미 destructure됨) 소스 ~4라인·타입 2필드·중복 진실원 제거.

#### 11. 최상위 신뢰도 구간 라벨이 "80~%"로 깨짐
- **파일/라인:** `web/components/performance/PhasePerformancePanel.tsx:84`
- **요지:** 구간 (lo=80, hi=101)에서 `b.hi >= 100 ? ""`가 틸드를 매달아 "80~%"로 렌더 — 다른 구간
  ("0~60%", "60~70%")과 불일치. "80%+" 또는 "80~100%"가 맞다.

#### 12. avg_return 정확히 0이 파란색(하락)으로 표시
- **파일/라인:** `web/components/performance/PhasePerformancePanel.tsx:60`
- **요지:** `b.avg_return > 0 ? text-up : text-down`에 0 분기가 없어 0.0%가 text-down(파랑/하락)으로
  렌더(최근 행 ~98라인도 동일). 한국 색 관례(빨강=상승/파랑=하락)가 load-bearing인 금융 UI에서
  보합인데 손실로 읽힘.

#### 13. confidence가 bool 수용 + 반올림 경계
- **파일/라인:** `scripts/phase_eval.py:117`
- **요지:** `round(float(conf)) if isinstance(conf, (int, float))`는 bool(int 하위클래스)을 수용해
  true/false가 1/0으로 저장됨. .5 값은 banker's rounding으로 인접 구간으로 넘어감.
- **재현:** phase.ts가 confidence를 boolean(또는 향후 0~1 스케일 — 문서화된 kimi prob 함정)으로
  직렬화하면 isinstance 통과해 1/0 저장 → (0,60)에 실데이터로 안착. 오늘 발생 가능성은 매우 낮음 —
  방어 nit(ai_click의 문서화된 잠재 함정과 동형).

## 검토했으나 제외한 항목
- `accuracy >= 50 → text-up`: 의도된 의미("코인플립보다 나음 = 좋음 색")로 판단 → 버그 아님.
- `DISCLAIMER` 리터럴 vs 시드 JSON: 바이트 단위 일치 확인.
- `_signal_and_window(span=1)`: 주석대로 D+1만 반환 — 정상.
- KV `phase:dates` SET TTL 없음(무한 증가): `/ai`에서 그대로 상속한 기존 동작 — 신규 회귀 아님.
- import/타입/KV 키 정합·crontab 멱등: 전부 확인 통과.
